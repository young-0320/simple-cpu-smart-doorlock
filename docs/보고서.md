# Simple CPU 기반 스마트 도어락 보고서

## I. 구조 설명

본 시스템은 Simple CPU 기반의 스마트 도어락으로, 물리 입력 처리 → CPU 프로그램 실행 → 물리 출력의 세 단계로 구성된다.

```
[물리 입력]
  pmod_key[9:0]  ─┐
  btn_input       │
  btn_confirm     ├─► input_handler ─► in_port[8:0]  ─┐
  btn_cancel      │                                   │
  btn_change      │                                   ▼
  btn_master      ┘                             ┌──────────┐
                                                │          ├──► bram_addr[11:0]
                           bram_rdata[31:0] ◄───┤ top_cpu  ├──► bram_wdata[31:0]
                                                │          ├──► bram_we
                                                └────┬─────┘
                                                     │ out_port[3:0]
                                                     ▼
                                              output_handler
                                               │          │
                                               ▼          ▼
                                           door_open   led[3:0]
```

- **클럭**: 보드 125MHz → `clk_wiz_0` → **10MHz** (`clk_cpu`) → 전 모듈 공급
- **리셋**: `ext_reset | ~clk_locked` = `sys_reset` (active-high, 동기)

## II. Instruction Format

### 2-1. ISA

| opcode | 니모닉   | Type | 동작                                                  |
| ------ | -------- | ---- | ----------------------------------------------------- |
| 0000   | LOAD     | M    | BRAM[addr] → ACC                                     |
| 0001   | STORE    | M    | ACC → BRAM[addr]                                     |
| 0010   | ADD      | M    | ACC + BRAM[addr] → ACC, ZERO_FLAG 업데이트           |
| 0011   | SUB      | M    | ACC - BRAM[addr] → ACC, ZERO_FLAG 업데이트           |
| 0100   | CMP      | M    | ACC와 BRAM[addr] 비교, ZERO_FLAG만 업데이트, ACC 보존 |
| 0101   | LOADI    | I    | immediate → ACC                                      |
| 0110   | ADDI     | I    | ACC + immediate → ACC, ZERO_FLAG 업데이트            |
| 0111   | CMPI     | I    | ACC와 immediate 비교, ZERO_FLAG만 업데이트, ACC 보존  |
| 1000   | JMP      | J    | PC ← addr                                            |
| 1001   | JZ       | J    | ZERO_FLAG = 1이면 PC ← addr                          |
| 1010   | JNZ      | J    | ZERO_FLAG = 0이면 PC ← addr                          |
| 1011   | NOP      | N    | 아무 동작 없음, PC만 증가                             |
| 1100   | OUT      | P    | ACC[3:0] → out_port[3:0]                             |
| 1101   | IN       | P    | 외부 입력 port 값을 ACC에 저장                        |
| 1110   | RESERVED | -    | NOP처럼 처리                                          |
| 1111   | EXT      | E    | funct[27:24]로 SHL/SHR/AND 디코딩                     |

**EXT의 function 구분**

| function | 니모닉 | 동작                      |
| -------- | ------ | ------------------------- |
| 0000     | SHL    | ACC를 shift left          |
| 0001     | SHR    | ACC를 shift right         |
| 0010     | AND    | ACC와 BRAM[addr] AND 연산 |

**각 type별 32-bit instruction 구조**

- **M type**: opcode(4) + reserved(16) + address(12)
- **I type**: opcode(4) + imm(28)
- **J type**: opcode(4) + reserved(16) + address(12)
- **N type**: opcode(4) + reserved(28)
- **P type**: opcode(4) + reserved(24) + port(4)
- **E type**: opcode(4) + funct(4) + reserved(12) + address(12)

```python
if cmd in EXT_FUNCT_MAP:
```


**I type**

```python
elif cmd in ["LOADI","ADDI","CMPI"]:

```

**J type**

```python
elif cmd in ["JMP", "JZ", "JNZ"]:
    opcode_bin = OPCODE_MAP[cmd]

    bin_32 = opcode_bin + reserved_bin + operand_bin
```

**N type**

```python
elif cmd in ["NOP"]:
    opcode_bin = OPCODE_MAP[cmd]
    reserved_bin = "0000000000000000000000000000"  # 28-bit 0
    bin_32 = opcode_bin + reserved_bin
```

**P type**

```python
elif cmd in ["OUT", "IN"]:
    opcode_bin = OPCODE_MAP[cmd]
    reserved_bin = "000000000000000000000000"  # 24-bit 0
    operand_bin = format(dec_val, '04b')

```

**BRAM 저장을 위한 임시 명령어 (특정 주소로 곧바로 이동)**

```python
if cmd == "ORG":
    new_address = parts[1]
    current_address = int(new_address, 16)
```

**2 pass 알고리즘**

- **pass1**: 라벨 주소록 수집 후, 해당 라벨을 현재주소(PC) 변경 없이 `label_map`에 매핑
- **pass2**: `label_map`을 참조하여 기계어 번역 진행

```python
# pass1
if len(parts) == 1 and clean_line.endswith(":"):
    label_name = clean_line[:-1]
    label_map[label_name] = current_address
else:
    current_address += 1

# pass2
elif cmd in OPCODE_MAP or cmd in EXT_FUNCT_MAP:
    trans_machine_code = assemble_line(cmd, operand)
    machine_codes[current_address] = trans_machine_code
    current_address += 1
```

### 2-3. doorlock.asm

#### 1) 비밀번호 입력 및 취소

**[확정]**
입력 비밀번호와 마스터키, 저장된 비밀번호의 반복적인 비교로 인한 지연을 줄이기 위해 unlock flag만 확인하여 곧바로 상태 변경

**[취소]**
SHR 기능을 통해 바로 이전에 입력된 한 자리 숫자 (4비트)를 지움

**[입력]**
입력된 숫자열 (BRAM-230)을 가져와서 SHL 후, 방금 입력된 한 자리 숫자(BRAM-231)를 더함

#### 2) 비밀번호 변경

비밀번호 변경을 위한 새로운 비밀번호 입력 시에도 (1)에서의 입력, 취소, 확정 기능 이용 가능

비밀번호 변경 시, 다음과 같은 알고리즘을 통해 새로운 비밀번호를 2번 입력하도록 한다.

1. 변경 버튼 입력
2. 변경 가능 상태인지 확인 (`CHECK_CAN_CHANGE`): unlock flag를 확인하여 문이 열려있는 상황에서만 비밀번호 변경 가능
3. 변경 모드: 입력, 취소, 확정 기능을 통해 새로운 비밀번호 입력
4. 새로운 비밀번호 재입력 (`ONE_MORE_CHECK1`, `ONE_MORE_CHECK2`, `UPDATE_MASK`): 새로운 비밀번호 첫 입력 시 비밀번호 변경 재확인용 공간(BRAM-243)에 저장. 재입력 시 BRAM-243과 비교 진행 후 재입력 FLAG 변경. 새로운 비밀번호 저장 및 그 길이에 맞는 마스크 변경

#### 3) 비밀번호 확인

기본적으로 unlock flag = 0인 경우에만 비교를 진행하여 불필요한 연산으로 인한 시간 지연을 줄임

비밀번호 입력 시, 다음과 같은 알고리즘을 통해 비밀번호 앞뒤에 다른 숫자를 입력하여도 문이 열리도록 하였다.

1. **입력된 비밀번호에서 유효한 숫자열 추출**: 비밀번호 매 '입력' 마다 입력 buffer(BRAM-230)과 현재 마스킹 데이터(BRAM-218)를 AND 연산하여 가장 최근 입력된 n자리를 알아냄 (n: 정답 비밀번호의 자리수)
2. **추출한 숫자열 비교**: 추출한 숫자열을 마스터키, 저장된 비밀번호와 비교
   - 정답: `OPEN_DOOR`
   - 오답: 실패 횟수 카운터(BRAM-232) +1 / 3회 오답 시 잠금 상태 유지. 잠금 상태 유지를 위한 타이머는 타이머를 위한 큰 수(BRAM-241)에서 0이 될 때까지 1을 지속적으로 뺄셈하는 방법으로 구현

## III. Module 및 IP 구성

### 3-1. CPU

#### 1) cpu_fsm.v

`cpu_fsm` 모듈은 CPU의 명령어 실행 주기를 4단계로 제어하는 유한 상태 기계(FSM)이다. `clk_enable` 신호로 일시정지가 가능하며, 현재 상태(`state`)를 나머지 모든 모듈이 공유한다.

**FSM 전이 구조**

```
FETCH → DECODE → EXECUTE → INCREMENT → FETCH → ...
```

상태는 매 클럭 상승 에지에서 순환 전이하며, reset 시 FETCH로 복귀한다. `clk_enable`이 비활성화된 동안에는 현재 상태를 유지함으로써 전체 CPU를 일시 정지시킨다.

```verilog
else if (clk_enable) begin
    case (state)
        FETCH:     state <= DECODE;
        DECODE:    state <= EXECUTE;
        EXECUTE:   state <= INCREMENT;
        INCREMENT: state <= FETCH;
        default:   state <= FETCH;
    endcase
end
```

각 상태의 역할은 다음과 같다.

| 상태      | 역할                                               |
| --------- | -------------------------------------------------- |
| FETCH     | `bram_addr ← pc_out`, BRAM에서 명령어 읽기 시작 |
| DECODE    | BRAM 출력을 IR에 래치, decoder 조합 논리 확정      |
| EXECUTE   | 메모리·I/O 명령어의 BRAM 주소 전환, STORE 실행    |
| INCREMENT | ACC·ZERO_FLAG·OUT_PORT 갱신, PC 증가             |

#### 2) pc.v

`pc` 모듈은 다음에 실행할 명령어의 BRAM 주소를 보관하는 12비트 프로그램 카운터이다. `top_cpu`에서 분기 로직이 계산한 `pc_next` 값을 받아 INCREMENT 상태에서만 갱신된다.

```verilog
assign pc_write = (state == `ST_INCREMENT);
```

`pc_write`와 `clk_enable`이 동시에 활성화된 클럭 상승 에지에서만 `pc_out ← pc_next`가 일어난다. 그 외 상태에서는 현재 값을 유지하여 FETCH 단계에서 동일한 주소로 BRAM을 읽을 수 있도록 한다.

```verilog
else if (clk_enable && pc_write) begin
    pc_out <= pc_next;
end
```

reset 시 `pc_out`은 0으로 초기화되어 BRAM 주소 0번지(프로그램 시작점)부터 실행이 재개된다.

#### 3) inst_reg.v

`inst_reg` 모듈은 BRAM에서 읽어온 32비트 명령어를 한 사이클 래치하는 명령어 레지스터이다. FETCH 단계에서 BRAM 출력이 안정화된 후, DECODE 상태 진입 시 `ir_we`가 활성화되어 명령어를 캡처한다.

```verilog
assign ir_we = clk_enable && (state == `ST_DECODE);
```

이후 EXECUTE·INCREMENT 단계 내내 `instr_out`이 고정된 상태로 decoder에 공급되므로, FSM이 다음 상태로 전이하는 동안 제어 신호가 흔들리지 않는다.

```verilog
else if (ir_we) begin
    instr_out <= instr_in;
end
```

reset 시 `instr_out`은 `32'hB0000000`으로 초기화된다. 상위 4비트 `4'b1011`은 `OP_NOP`에 해당하여, 리셋 직후 의도치 않은 연산이 실행되는 것을 방지한다.

#### 4) decoder.v

`decoder` 모듈은 32비트 명령어를 해석하여 CPU 전체의 데이터패스 제어 신호를 생성하는 순수 조합 논리 블록이다. 클럭이 없으며, `inst_reg`의 `instr_out`이 변경되는 즉시 출력이 갱신된다.

**필드 분리**

```verilog
wire [3:0] opcode = instruction[31:28];
wire [3:0] funct  = instruction[27:24];  // OP_EXT 전용

assign imm  = instruction[27:0];   // 28비트 즉시값
assign addr = instruction[11:0];   // BRAM/분기 주소
assign port = instruction[3:0];    // I/O 포트 번호
```

**1단계 — opcode 디코딩**

`case (opcode)` 블록에서 16개의 opcode를 분류하여 명령어 종류 플래그(`is_load`, `is_store`, `is_alu_mem`, `is_alu_imm`, `is_ext_imm`, `is_jump`, `is_in`, `is_out`)와 ALU 연산 코드(`alu_ctrl`), zero 플래그 갱신 여부(`zero_we`), 분기 종류(`jump_type`)를 결정한다.

**2단계 — OP_EXT 추가 디코딩**

opcode가 `4'b1111`(`OP_EXT`)인 경우 `funct[27:24]`를 기준으로 2차 디코딩을 수행한다.

```verilog
`OP_EXT: begin
    case (funct)
        `EXT_SHL: begin is_ext_imm = 1'b1; alu_ctrl = `ALU_SHL; zero_we = 1'b1; end
        `EXT_SHR: begin is_ext_imm = 1'b1; alu_ctrl = `ALU_SHR; zero_we = 1'b1; end
        `EXT_AND: begin is_alu_mem = 1'b1; alu_ctrl = `ALU_AND; zero_we = 1'b1; end
        default: ;
    endcase
end
```

래치 생성을 방지하기 위해 `always @(*)` 블록 최상단에서 모든 출력 신호를 안전 기본값으로 초기화한 후 case 문을 실행한다.

#### 5) accumulator.v

`accumulator` 모듈은 CPU의 유일한 범용 레지스터인 32비트 누산기(ACC)이다. 연산 결과, LOAD 데이터, IN 포트 값 등 다양한 소스가 `top_cpu`의 MUX를 거쳐 `acc_in`으로 입력되며, `acc_we`가 활성화된 클럭 상승 에지에서 저장된다.

```verilog
else if (acc_we) begin
    acc_out <= acc_in;
end
```

`acc_we` 활성화 조건은 `top_cpu`의 조합 논리에서 결정되며, INCREMENT 상태에서만 갱신된다. CMP 명령어는 `zero_flag`만 갱신하고 ACC를 변경하지 않으므로 `acc_we`를 활성화하지 않는다.

```verilog
if (is_alu_imm || is_alu_mem || is_ext_imm) begin
    if (alu_ctrl != `ALU_CMP) begin
        acc_in = alu_result;
        acc_we = 1'b1;
    end
end
```

#### 6) alu.v

`alu` 모듈은 `acc_out`과 `operand_in`을 받아 7가지 연산을 수행하는 순수 조합 논리 블록이다. `alu_ctrl`에 따라 `alu_result`와 zero 플래그(`zero_result`)를 즉시 출력한다. `top_cpu`에서 `alu_operand` MUX를 통해 피연산자 소스(즉시값·시프트량·BRAM 데이터)가 선택된 후 `operand_in`으로 전달된다.

```verilog
assign alu_operand = is_alu_imm ? {4'b0000, imm}  :
                     is_ext_imm ? {20'b0, addr}    :
                                  bram_rdata;
```

| `alu_ctrl` | 연산                    | `zero_result` 조건 |
| ------------ | ----------------------- | -------------------- |
| ALU_ADD      | `acc + operand`       | 결과 == 0            |
| ALU_SUB      | `acc − operand`      | 결과 == 0            |
| ALU_CMP      | acc 변경 없음           | `acc == operand`   |
| ALU_PASS     | `operand`             | `operand == 0`     |
| ALU_SHL      | `acc << operand[4:0]` | 결과 == 0            |
| ALU_SHR      | `acc >> operand[4:0]` | 결과 == 0            |
| ALU_AND      | `acc & operand`       | 결과 == 0            |

CMP는 acc를 그대로 통과시키되 두 값이 같을 때 `zero_result`를 1로 설정하여, 후속 JZ/JNZ 분기의 기준 플래그로 활용된다.

#### 7) top_cpu.v

`top_cpu` 모듈은 `cpu_fsm`, `pc`, `inst_reg`, `decoder`, `accumulator`, `alu` 여섯 서브모듈을 통합하는 CPU 최상위 모듈이다. BRAM 인터페이스, 9비트 입력 포트, 4비트 출력 포트, 디버그 출력을 외부에 노출한다.

**데이터패스 전체 흐름**

```
FETCH     : bram_addr ← pc_out  →  BRAM 명령어 출력 대기
DECODE    : ir_we 활성화  →  instr_out ← bram_rdata  →  decoder 신호 확정
EXECUTE   : 메모리/I/O 명령어: bram_addr ← addr, STORE: bram_we 활성화
INCREMENT : ACC·ZERO_FLAG·OUT_PORT 갱신, PC ← pc_next
```

**1단계 — BRAM 주소 및 쓰기 제어**

FETCH·DECODE·INCREMENT에서는 `bram_addr = pc_out` (명령어 페치). EXECUTE에서는 LOAD·STORE·ALU_MEM 명령어일 때 `bram_addr = addr`로 전환하여 데이터 영역에 접근한다.

```verilog
`ST_EXECUTE: begin
    if (is_load || is_store || is_alu_mem)
        bram_addr = addr;
    if (is_store)
        bram_we = clk_enable;
end
```

**2단계 — ACC 및 ZERO_FLAG 갱신**

INCREMENT 상태에서 명령어 종류에 따라 `acc_in` 소스와 `acc_we`를 결정한다.

```
if (is_load)                                    { acc_in = bram_rdata;     acc_we = 1; }
else if (is_in)                                 { acc_in = selected_input; acc_we = 1; }
else if (is_alu_imm || is_alu_mem || is_ext_imm)
    if (alu_ctrl != `ALU_CMP)                  { acc_in = alu_result;     acc_we = 1; }
```

`zero_flag`는 `zero_we`가 활성화된 연산에서만 `alu_zero_result`로 갱신된다.

**3단계 — PC next 로직**

기본값은 `pc_out + 1`이며, `is_jump`가 활성화된 경우 `jump_type`과 `zero_flag`에 따라 분기 주소 `addr`로 변경된다.

```verilog
`JMP_UNCOND: pc_next = addr;
`JMP_JZ:     pc_next = zero_flag  ? addr : pc_out + 1;
`JMP_JNZ:    pc_next = !zero_flag ? addr : pc_out + 1;
```

**4단계 — IN 포트 선택**

IN 명령어의 `operand[3:0]`(port)에 따라 9비트 `in_port`를 6개의 논리 채널로 매핑한다.

```verilog
`IN_PORT_NUM:     selected_input = {28'd0, in_port[3:0]};  // 숫자 입력
`IN_PORT_BTN_IN:  selected_input = {31'd0, in_port[4]};    // 입력 확정
`IN_PORT_BTN_CFM: selected_input = {31'd0, in_port[5]};    // 전체 확정
`IN_PORT_BTN_CAN: selected_input = {31'd0, in_port[6]};    // 취소
`IN_PORT_BTN_PW:  selected_input = {31'd0, in_port[7]};    // 비밀번호 변경
`IN_PORT_MST_KEY: selected_input = {31'd0, in_port[8]};    // 마스터키
```

OUT 명령어는 INCREMENT에서 `acc_out[3:0]`을 `out_port`에 직접 래치하며, 초기값 `OUT_STATE_CLOSED`(`4'b0000`)는 도어락 잠김 상태를 의미한다.

### 3-2. Top Controller

#### 1) debouncer.v

`debouncer` 모듈은 외부 비동기 버튼과 스위치 입력의 채터링을 제거하고, 안정된 레벨 신호(`btn_level`)와 상승 에지 펄스(`btn_pulse`)를 출력한다. `input_handler` 내부에서 PMOD 키 10개와 버튼/스위치 5개에 대해 총 15번 인스턴스화된다. 동작은 세 단계로 구성된다.

**1단계 — 2-Stage Synchronizer**

```verilog
sync_0 <= btn_in;
sync_1 <= sync_0;
```

외부 비동기 신호(`btn_in`)를 클럭 도메인으로 가져올 때 발생할 수 있는 metastability를 방지하기 위해 두 개의 D 플립플롭(`sync_0`, `sync_1`)을 직렬로 연결한다. 이후 모든 로직은 `sync_1`을 기준으로 동작한다.

**2단계 — 카운터 기반 채터링 제거**

```verilog
if (sync_1 == btn_stable) begin
    cnt <= 32'd0;
end else begin
    if (cnt == COUNT_MAX - 1) begin
        btn_stable <= sync_1;
        cnt <= 32'd0;
    end else begin
        cnt <= cnt + 1'b1;
    end
end
```

`sync_1`이 현재 안정 상태(`btn_stable`)와 다를 경우 카운터를 증가시키고, `COUNT_MAX - 1`에 도달하면 비로소 `btn_stable`을 새로운 상태로 갱신한다. `sync_1`이 `COUNT_MAX` 클럭 동안 일정하게 유지되어야만 상태 변화로 인정하는 방식이다. `COUNT_MAX`는 `define.vh`에서 `CLK_FREQ`와 `DEBOUNCE_MS`로부터 자동 계산되며, 10MHz 클럭 기준 20ms에 해당하는 200,000 클럭이다. `sync_1`이 도중에 다시 `btn_stable`과 같아지면 카운터는 즉시 0으로 리셋된다.

**3단계 — 출력 생성 및 상승 에지 검출**

```verilog
btn_stable_d <= btn_stable;
btn_level    <= btn_stable;
// 이전 사이클은 0이었고 현재 사이클은 1일 때만 펄스 발생 (Rising Edge)
btn_pulse    <= btn_stable & ~btn_stable_d;
```

`btn_stable`을 1클럭 지연한 `btn_stable_d`를 두어, `btn_stable & ~btn_stable_d` 조건으로 상승 에지를 검출한다. 이 값이 `btn_pulse`로 출력되며, 버튼을 누르는 순간 정확히 1클럭만 High가 된다. `btn_level`은 `btn_stable`을 그대로 출력한다.

#### 2) input_handler.v

`input_handler`는 보드로부터의 외부 입력을 받아 처리한다. 출력 `in_port`는 `top_doorlock` 모듈에서 인스턴스화를 통해 cpu 모듈로 입력된다.

| 신호            | 방향 | 비트폭 | 설명                     |
| --------------- | ---- | ------ | ------------------------ |
| `clk`         | 입력 | 1      | 10MHz CPU 클럭           |
| `reset`       | 입력 | 1      | 시스템 리셋              |
| `pmod_key`    | 입력 | 10     | PMOD 숫자 입력 (one-hot) |
| `btn_input`   | 입력 | 1      | BTN1: 자릿수 입력 확정   |
| `btn_confirm` | 입력 | 1      | BTN2: 비밀번호 전체 확정 |
| `btn_cancel`  | 입력 | 1      | BTN3: 마지막 자리 취소   |
| `btn_change`  | 입력 | 1      | SW2: 비밀번호 변경 모드  |
| `btn_master`  | 입력 | 1      | SW3: 마스터키            |
| `in_port`     | 출력 | 9      | CPU IN 명령어 입력 버스  |

외부 입력은 숫자 입력과 버튼 입력의 두 종류로 구분된다. 숫자 입력은 PMOD 커넥터(JE/JD)에 연결된 10비트 신호로 숫자 0~9를 one-hot 방식으로 표현한다. 본 구현에서는 점퍼선을 GND/VCC 간 전환하는 방식으로 입력하였다. 버튼 입력은 온보드 버튼 BTN1~BTN3과 슬라이드 스위치 SW2~SW3을 사용한다.

| 신호            | 핀         | 역할               |
| --------------- | ---------- | ------------------ |
| `btn_input`   | BTN1 (P16) | 입력 확정          |
| `btn_confirm` | BTN2 (K19) | 비밀번호 전체 확정 |
| `btn_cancel`  | BTN3 (Y16) | 마지막 입력 취소   |
| `btn_change`  | SW2 (W13)  | 비밀번호 변경 모드 |
| `btn_master`  | SW3 (T16)  | 마스터키 입력      |

`input_handler`는 버튼 채터링을 제거하는 디바운서와, 10비트 one-hot 신호를 4비트 숫자로 변환하는 우선순위 인코더를 포함한다.

```verilog
reg [3:0] key_num;
always @(*) begin
    case (key_level)
        10'b0000000001: key_num = 4'd0;
        10'b0000000010: key_num = 4'd1;
        10'b0000000100: key_num = 4'd2;
        10'b0000001000: key_num = 4'd3;
        10'b0000010000: key_num = 4'd4;
        10'b0000100000: key_num = 4'd5;
        10'b0001000000: key_num = 4'd6;
        10'b0010000000: key_num = 4'd7;
        10'b0100000000: key_num = 4'd8;
        10'b1000000000: key_num = 4'd9;
        default:        key_num = 4'd0;
    endcase
end
```

`pmod_key[9:0]`는 숫자 하나에 해당하는 비트 하나만 1인 one-hot 형식이다. 디바운싱된 레벨 신호(`key_level`)를 case문으로 우선순위 인코딩하여 0부터 9까지의 4비트 숫자값(`key_num`)으로 변환한다. 점퍼선이 꽂혀 있는 동안 해당 숫자가 유지되므로, BTN1을 누르는 순간의 `key_num` 값이 CPU에 읽힌다.

```verilog
always @(posedge clk) begin
    if (reset) begin
        latch_input   <= 1'b0; cnt_input   <= 6'd0;
        latch_confirm <= 1'b0; cnt_confirm <= 6'd0;
        latch_cancel  <= 1'b0; cnt_cancel  <= 6'd0;
        latch_change  <= 1'b0; cnt_change  <= 6'd0;
        latch_master  <= 1'b0; cnt_master  <= 6'd0;
    end
    else begin
        // btn_input
        if (pulse_input) begin
            latch_input <= 1'b1;
            cnt_input   <= LATCH_HOLD;
        end
        else if (cnt_input != 6'd0) begin
            cnt_input <= cnt_input - 1'b1;
        end
        else begin
            latch_input <= 1'b0;
        end
    // ...
    end
end
```

버튼/스위치의 `btn_pulse`는 1클럭짜리 펄스이다. CPU의 MAIN_WAIT 루프는 한 바퀴에 약 28사이클이므로, 1클럭 펄스만으로는 CPU가 IN 명령어를 실행하는 타이밍에 읽지 못할 수 있다. 이를 해결하기 위해 펄스 발생 시 래치를 SET하고 64클럭(`LATCH_HOLD = 63`) 동안 유지한 뒤 자동으로 CLR하는 카운터 기반 래치를 사용한다. CPU는 MAIN_WAIT 루프를 1~2바퀴 도는 동안 반드시 해당 래치를 읽을 수 있다. pulse가 발생하면 latch는 1이 되고 counter는 63이 된다. 매 클럭마다 counter는 1씩 감소하며 counter가 0이 되면 latch도 0으로 초기화되며 재시작이 가능해진다. confirm, cancel, change, master 버튼도 `btn_input`의 래치 로직과 동일한 방식으로 처리하였다.

9비트 출력 `in_port`에는 입력된 비밀번호 값과 래치 처리를 거친 `latch_input`, `latch_confirm`, `latch_cancel`, `latch_change`, `latch_master`가 순서대로 저장된다. 이는 `top_doorlock` 모듈을 거쳐 `top_cpu` 모듈의 입력으로 전달된다.

#### 3) output_handler.v

`output_handler` 모듈은 CPU의 OUT 명령어가 기록한 출력 `out_port`를 받아 물리적 출력 신호로 변환하는 모듈이다.

| 신호          | 방향 | 비트폭 | 설명                             |
| ------------- | ---- | ------ | -------------------------------- |
| `clk`       | 입력 | 1      | 10MHz CPU 클럭                   |
| `reset`     | 입력 | 1      | 시스템 리셋                      |
| `out_port`  | 입력 | 4      | CPU OUT 명령어 출력 버스         |
| `door_open` | 출력 | 1      | 도어락 솔레노이드 제어 (JD, V18) |
| `led`       | 출력 | 4      | 온보드 LED LD3~LD0               |

#### 4) top_doorlock.v

`top_doorlock` 모듈은 클럭, BRAM, input_handler, CPU, output_handler를 인스턴스화하여 도어락의 전체 동작을 제어한다.

| 신호            | 방향 | 비트폭 | 설명                        |
| --------------- | ---- | ------ | --------------------------- |
| `clk`         | 입력 | 1      | 보드 125MHz 클럭            |
| `ext_reset`   | 입력 | 1      | 시스템 리셋                 |
| `pmod_key`    | 입력 | 10     | PMOD 숫자 입력 (one-hot)    |
| `btn_input`   | 입력 | 1      | BTN1: 자릿수 입력 확정      |
| `btn_confirm` | 입력 | 1      | BTN2: 비밀번호 전체 확정    |
| `btn_cancel`  | 입력 | 1      | BTN3: 마지막 자리 취소      |
| `btn_change`  | 입력 | 1      | SW2: 비밀번호 변경 모드     |
| `btn_master`  | 입력 | 1      | SW3: 마스터키               |
| `door_open`   | 출력 | 1      | 도어락 솔레노이드 제어 신호 |
| `led`         | 출력 | 4      | 도어락 상태 출력 LED        |

**Clocking Wizard IP**

```verilog
wire clk_cpu;
wire clk_locked;

clk_wiz_0 u_clk_wiz (
    .clk_out1 (clk_cpu),
    .locked   (clk_locked),
    .clk_in1  (clk)
);
```

Clock IP는 보드 입력 125MHz를 CPU 주파수 10MHz로 분주한다. 이를 CPU 및 handler 모듈에 연결하기 위해 wire `clk_cpu`를 사용하였다.

```verilog
wire sys_reset;
assign sys_reset = ext_reset | ~clk_locked;
```

`sys_reset` wire는 PLL이 안정화되기 전 1을 유지하며 모든 하위 모듈이 리셋 상태이도록 유지한다. 이를 통해 PLL이 lock되고 `ext_reset`이 해제되어야 동작이 시작되도록 하였다.

**BRAM IP**

```verilog
wire [11:0] bram_addr;
wire [31:0] bram_wdata;
wire [31:0] bram_rdata;
wire        bram_we;

blk_mem_gen_0 u_bram (
    .clka  (clk_cpu),
    .ena   (1'b1),
    .wea   ({bram_we}),
    .addra (bram_addr),
    .dina  (bram_wdata),
    .douta (bram_rdata)
);
```

BRAM IP는 32비트 폭, 4096 깊이의 Single Port RAM IP이다. 주소 `bram_addr`, 쓰기 데이터 `bram_wdata`, 쓰기 활성화 `bram_we`, 읽기 데이터 `bram_rdata`는 모두 `top_cpu`와 직접 연결된다. `ena`는 상시 1로 고정하여 항상 활성 상태를 유지한다. COE 파일로 초기값이 로드된다.

**input_handler**

```verilog
wire [8:0] in_port;

input_handler u_input_handler (
    .clk         (clk_cpu),
    .reset       (sys_reset),
    .pmod_key    (pmod_key),
    .btn_input   (btn_input),
    .btn_confirm (btn_confirm),
    .btn_cancel  (btn_cancel),
    .btn_change  (btn_change),
    .btn_master  (btn_master),
    .in_port     (in_port)
);
```

**top_cpu**

```verilog
wire [3:0] out_port;

top_cpu u_cpu (
    .clk        (clk_cpu),
    .reset      (sys_reset),
    .clk_enable (clk_enable),
    .bram_rdata (bram_rdata),
    .bram_addr  (bram_addr),
    .bram_wdata (bram_wdata),
    .bram_we    (bram_we),
    .in_port    (in_port),
    .out_port   (out_port)
);
```

**output_handler**

```verilog
output_handler u_output_handler (
    .clk       (clk_cpu),
    .reset     (sys_reset),
    .out_port  (out_port),
    .door_open (door_open),
    .led       (led)
);
```

`output_handler`는 CPU에서 출력된 `out_port`를 받아 그 상태를 바탕으로 도어락 제어 신호 `door_open`과 4비트 LED 상태를 출력한다. 이는 Zybo Z7-20 보드의 XDC 파일을 통해 보드의 LED 포트와 연결되어 사용자가 현재 상태를 확인할 수 있도록 한다.

### 3-3. IP`

#### 1) BRAM

본 설계에서는 명령어 및 도어락의 비밀번호를 저장하기 위해 BRAM을 사용하였다. Simple CPU의 명령어 및 도어락 동작을 위한 명령어와 비밀번호 저장 공간을 위하여 BRAM의 크기는 32-bit width, 4096-word depth로 설정하였다.

기본 설정은 변경 없이 유지하였으며, width는 32비트, depth는 4096-word로 설정하였다. 기존에 설계한 assembly에 맞게 생성한 COE 파일을 BRAM의 초기 메모리로 설정하였다.

#### 2) Clock

클럭의 경우 보드의 주파수 125MHz를 동일하게 CPU에 넣을 경우 클럭이 너무 빨라 정상적으로 작동하지 못하기 때문에 Clocking Wizard IP를 통해 125MHz 주파수를 10MHz로 분주하였다.

PLL 방식으로 입력 주파수는 125MHz이며, 출력 주파수를 10MHz로 설정하였다. 본 설계에서는 ILA를 사용하지 않으므로 PLLE2의 compensation을 BUFIN으로 변경하지 않고 BRAM의 최적화 상태로 유지하였다.

## IV. 결과 및 시연

본 프로젝트에서는 Simple CPU 기반 스마트 도어락 시스템을 구현하고, FPGA 보드를 이용하여 주요 기능이 정상적으로 동작하는지 확인하였다. 입력은 PMOD를 이용한 one-hot 방식으로 구성하였으며, 사용자가 입력한 숫자에 따라 도어락 열림, 입력 취소, 오류 횟수 표시, 입력 제한, 마스터키 입력, 비밀번호 변경 기능이 동작하도록 하였다. 시연은 실제 기능 동작을 중심으로 진행하였으며, CPU 내부 구조와 코드 동작 방식은 발표에서 별도로 설명하였다.

먼저 보드를 리셋하여 초기 상태를 확인하였다. 리셋 후 아무 LED도 켜져 있지 않은 상태를 도어락이 닫혀 있는 기본 상태로 설정하였다. 이 상태에서 숫자 입력이 들어오면 도어락 동작이 시작되며, 초기 입력 대기 상태가 정상적으로 유지되는 것을 확인하였다.

다음으로 초기 비밀번호 입력 기능을 확인하였다. 초기 비밀번호는 1, 2, 3, 4로 설정하였고, 해당 순서대로 숫자를 입력하였을 때 Open LED가 점등되는 것을 확인하였다. 이는 저장된 비밀번호와 입력된 숫자열이 일치할 경우 도어락이 정상적으로 열림 상태로 전환됨을 의미한다. 이후 실제 문을 닫는 상황을 reset 버튼으로 표현하였으며, reset 버튼을 누르면 켜져 있던 Open LED가 꺼지면서 다시 닫힘 상태가 되는 것을 확인하였다.

또한 입력 흐름 중 비밀번호가 포함되는 경우에도 도어락이 열리는지 확인하였다. 저장된 비밀번호가 1234라고 가정하고, 전체 입력을 5, 6, 1, 2, 3, 4, 7, 8 순서로 진행하였다. 이 경우 전체 입력값은 비밀번호와 완전히 같지 않지만, 입력 흐름 중 1, 2, 3, 4가 연속으로 포함되어 있다. 시연 결과 해당 입력에서도 Open LED가 점등되었으며, 입력 숫자열 중 저장된 비밀번호가 연속된 부분 문자열로 포함되면 도어락이 열리는 기능이 정상적으로 동작함을 확인하였다.

입력 취소 기능도 확인하였다. 먼저 1, 2를 입력한 뒤 잘못된 숫자인 9를 입력하였다. 이후 취소 버튼을 눌러 마지막에 입력된 9가 삭제되도록 하였고, 이어서 3, 4를 입력하였다. 결과적으로 입력된 숫자열은 1, 2, 3, 4가 되었으며, Open LED가 점등되는 것을 확인하였다. 이를 통해 취소 버튼을 누르면 가장 최근 입력된 숫자 한 자리가 정상적으로 삭제되고, 이후 입력이 올바르게 이어지는 것을 확인하였다.

비밀번호 오류 횟수에 따른 LED 표시 기능도 시연하였다. 먼저 틀린 비밀번호인 1, 1, 1, 1을 입력하였을 때 실패 횟수가 1회로 증가하고 6번 LED에 빨간색 LED가 점등되는 것을 확인하였다. 이어서 다시 틀린 비밀번호인 2, 2, 2, 2를 입력하였을 때 실패 횟수가 2회로 증가하고 5번 LED와 6번 LED에 빨간색 LED가 모두 점등되는 것을 확인하였다. 이를 통해 연속 오류 횟수가 누적되며, 오류 횟수에 따라 LED 출력 상태가 달라지는 것을 확인할 수 있었다.

세 번째 오류 입력에서는 3, 3, 3, 3을 입력하여 연속 오류 3회 상태를 만들었다. 이때 기존 Red LED에 추가로 외부 Red LED까지 점등되어 총 3개의 LED가 켜지는 것을 확인하였다. 또한 연속 3회 오류가 발생하면 보안 기능으로 입력 제한 상태에 진입하도록 하였으며, 해당 상태에서는 숫자를 입력해도 도어락이 반응하지 않는 것을 확인하였다. 입력 제한 시간이 지난 후에는 다시 비밀번호 입력이 가능해지는 것을 확인하였다. 이를 통해 연속 오류 발생 시 경고 LED 표시와 입력 제한 기능이 정상적으로 동작함을 확인하였다.

마스터키 기능도 확인하였다. 마스터키는 일반 비밀번호와 별도로 저장된 번호이며, 본 시연에서는 9999를 마스터키로 설정하였다. 마스터키인 9, 9, 9, 9를 입력하였을 때 일반 비밀번호 입력 과정과 관계없이 Open LED가 즉시 점등되는 것을 확인하였다. 이를 통해 BRAM에 저장된 마스터키와 입력값이 일치할 경우 도어락이 즉시 열리는 기능이 정상적으로 동작함을 확인하였다.

마지막으로 비밀번호 변경 기능을 시연하였다. 스위치를 올려 도어락을 change mode로 전환한 뒤, 새 비밀번호를 8자리로 설정하였다. 본 시연에서는 새 비밀번호를 1, 2, 3, 4, 5, 6, 7, 8로 입력하였고, 확인을 위해 동일한 비밀번호를 한 번 더 입력하였다. 두 입력이 일치하면 비밀번호가 변경되도록 하였으며, 이후 변경된 비밀번호가 정상적으로 적용되었는지 확인하였다.

비밀번호 변경 후에는 기존 비밀번호였던 1, 2, 3, 4를 입력하여 도어락이 열리지 않는 것을 확인하였다. 이후 새로 설정한 8자리 비밀번호인 1, 2, 3, 4, 5, 6, 7, 8을 입력하였을 때 Open LED가 점등되는 것을 확인하였다. 이를 통해 기존 비밀번호가 더 이상 유효하지 않고, 새로 저장된 비밀번호로 도어락이 정상적으로 열리는 것을 확인하였다.

전체 시연 결과, 초기 비밀번호 입력, 입력 흐름 중 비밀번호 포함 판단, 입력 취소, 오류 횟수에 따른 LED 표시, 연속 오류 3회 발생 시 입력 제한, 마스터키 입력, 비밀번호 변경 기능이 모두 정상적으로 동작하였다. 따라서 본 프로젝트에서 목표로 한 스마트 도어락의 주요 기능이 FPGA 보드 상에서 정상적으로 구현되었음을 확인할 수 있었다.

## V. 결론

본 프로젝트에서는 직접 설계한 Simple CPU를 FPGA 보드(Zybo Z7-20)에 구현하고, 이를 기반으로 스마트 도어락 시스템을 완성하였다. CPU의 ISA 설계부터 어셈블러 개발, RTL 구현, 그리고 실제 하드웨어 동작 검증까지 디지털 시스템의 전체 설계 흐름을 직접 수행하였다는 점에서 의의가 있다.

하드웨어 측면에서는 4단계 FSM(FETCH–DECODE–EXECUTE–INCREMENT) 기반 CPU 파이프라인을 설계하고, BRAM을 명령어 메모리와 데이터 메모리로 통합 운용하는 구조를 구현하였다. 채터링 제거를 위한 디바운서와 래치 기반 입력 홀드 회로를 통해 물리 버튼 입력의 신뢰성을 확보하였으며, Clocking Wizard IP를 활용한 클럭 분주로 안정적인 동작 환경을 구성하였다.

소프트웨어 측면에서는 16종의 명령어를 지원하는 ISA를 정의하고, 2-pass 알고리즘 기반 어셈블러(assembler.py)를 직접 제작하여 어셈블리 코드에서 COE 파일까지 자동 변환하는 환경을 구축하였다. 비밀번호 입력, 취소, 확정, 비교, 변경, 마스터키, 오류 횟수 제한 등 도어락의 핵심 기능을 모두 어셈블리 수준에서 구현하였으며, AND 마스킹을 활용한 부분 문자열 비교 방식으로 입력 흐름 중 비밀번호를 유연하게 인식하는 알고리즘을 적용하였다.

시연을 통해 초기 비밀번호 인식, 부분 문자열 포함 판단, 입력 취소, 연속 오류에 따른 경고 LED 표시 및 입력 제한, 마스터키, 비밀번호 변경 등 계획한 모든 기능이 실제 보드에서 정상 동작함을 확인하였다.

향후 개선 방향으로는 파이프라인 구조 도입을 통한 처리 성능 향상, 인터럽트 기반 입력 처리 방식 적용, 그리고 다자리 비밀번호 입력 시 실시간 마스킹 갱신 최적화 등을 고려할 수 있다. 본 프로젝트는 소규모 CPU 아키텍처와 임베디드 소프트웨어, 물리 인터페이스를 하나의 시스템으로 통합하는 과정을 직접 경험함으로써, 디지털 시스템 설계의 전반적인 이해를 높이는 데 기여하였다.
